# LVGL 代码设计思路及代码规范

## 一、总体设计思路

### 1. UI 开发工具与版本要求

- **推荐工具**：使用 GUI-Guider-1.9.1-GA 或更高版本进行 UI 设计
- **核心框架**：基于 LVGL 版本 8 构建用户界面，确保兼容性和稳定性

### 2. 文件架构组织

项目采用模块化的三层文件结构，确保代码清晰可维护：

```c
custom/          # 用户自定义文件夹，内置经过验证的通用模板，通常无需替换
generated/       # 软件自动生成的页面文件，包含字体、图片等资源
                 # 在存储空间充足的情况下，可直接使用该目录内容
```



### 3. 逻辑通信机制

- **架构设计**：采用前后端分离模式，UI 与业务逻辑通过消息队列进行解耦通信
- **双向数据流**：
  - **下行通道**：主逻辑 → LVGL，发送页面切换指令和状态更新
  - **上行通道**：LVGL → 主逻辑，通过 `event_init.c` 中的事件处理函数，将用户交互事件发送至主逻辑接收线程处理

## 二、代码规范细则

### 1. 命名规范

以 `0310ui` 项目为例，建立统一的命名体系：

- **一致性原则**：页面名称与控件名称保持严格一致
- **核心价值**：确保页面切换和事件绑定实现零重写，同时使每个控件的功能用途一目了然

### 2. 自定义文件管理

`custom` 文件夹作为自定义代码的核心存储区，结构规划如下：

```c
custom/
├── custom.c              # GUI-Guider 生成的骨架文件（保持稳定，不随意替换）
├── language_manager.c    # 多语言管理核心模块
└── ...                   # 其他业务自定义文件
```



### 3. 核心文件详解

#### custom.c：页面导航控制中心

- **核心功能**：维护全页面函数接口的映射数组，实现统一的页面调度
- **设计理念**：对原生页面切换函数进行高层抽象和业务封装
- **系统角色**：作为 LVGL 页面切换的中枢调度器，所有页面跳转逻辑均通过该文件进行统一管理

#### tanslation.c：国际化语言支持引擎

- **功能定位**：实现四国语言（含繁体中文）的动态文本翻译
- **运行机制**：基于函数映射表架构，在页面切换完成后自动触发对应页面的文本本地化

##### 关键实施要点：

1. **映射表维护**：必须保持控件命名的严格一致性。当控件结构发生变更时，需同步更新映射表中的对应控件引用
2. **翻译管理**：以中文文本为基准语言，修改时只需同步更新同一翻译数组中的其他语言版本

## 总结与最佳实践

### 架构优势体现

- **模块化设计**：各功能模块边界清晰、职责单一，极大提升团队协作效率和系统可维护性
- **扩展性保障**：通过映射表机制，新增页面只需简单注册即可，无需触及核心逻辑代码
- **国际化支持**：内置的多语言框架为产品全球化部署提供坚实的技术基础
- **系统健壮性**：消息队列配备智能重试机制，有效保障系统启动阶段的通信可靠性

### 开发实施建议

1. **规范遵循**：严格执行命名规范，确保页面和控件标识的全系统一致性
2. **同步维护**：UI 结构修改时，必须同步更新语言映射表和页面映射表
3. **接口统一**：优先使用 custom.c 提供的统一导航接口，避免直接调用底层 LVGL 函数
4. **兼容性保障**：建立定期检查机制，确保生成文件与自定义文件的版本兼容性

### 系统维护要点

- **消息队列管理**：针对 LVGL 线程优先级较低的特点，在系统启动初期采用安全发送函数，规避因队列初始化延迟导致的通信失败
- **资源优化管理**：在资源受限环境下，对 generated 文件夹中的资源文件实施精细化管理，建立定期清理机制
- **启动流程优化**：通过依赖检查和延时等待机制，确保消息队列在业务逻辑启用前完成初始化
- **错误处理体系**：为所有队列操作建立完整的错误处理和日志记录机制，提升问题排查效率
- **性能监控体系**：
  - 实时监控消息队列的运行状态，快速识别和处理队列阻塞或溢出问题
  - 启用 `LV_USE_MEM_MONITOR` 宏并结合 custom.c 文件中的 `monkey_test` 函数，对 LVGL 进行高频率运行压力测试，检测内存碎片产生情况
- **错误追踪能力**：充分利用 lv_conf.h 文件中的错误打印功能，结合芯片栈回溯工具通过串口输出，实现问题代码的精确定位

### 风险防控策略

- **智能重试机制**：建立合理的重试策略和超时控制，在系统响应速度和通信可靠性之间取得最佳平衡
- **链路健康检查**：在关键业务流程中嵌入队列状态检查点，确保持续的通信链路畅通



### 三、补充要求与高级实践

#### 1. 内存与性能管理规范

**a) 对象生命周期管理**

- **创建原则**：所有动态创建的 LVGL 对象（`lv_obj_t*`）必须在 `custom.c` 的页面卸载函数或事件回调中明确其所有者。
- **销毁策略**：页面切换时，必须调用 `lv_obj_del` 或 `lv_obj_clean` 彻底清理非共享型控件，防止内存泄漏。对于频繁切换的页面，考虑对象池技术。

**b) 样式管理**

- **全局样式**：在 `custom.c` 中定义项目级的全局样式（如默认字体、主题色），确保 UI 统一性。
- **局部样式**：页面特有样式应在页面初始化函数中创建，并在页面卸载时销毁。避免使用过多的 `lv_style_set_...` 函数动态修改，应使用静态样式变量。



#### 2. 事件处理高级规范

**a) 事件回调函数设计**

```c
// 推荐的事件回调结构
static void event_callback(lv_event_t * e) {
    lv_event_code_t code = lv_event_get_code(e);
    lv_obj_t * obj = lv_event_get_target(e);
    
    switch(code) {
        case LV_EVENT_CLICKED:
            // 获取自定义数据
            custom_data_t * data = lv_event_get_user_data(e);
            if(data && data->is_valid) {
                handle_button_click(data->param);
            }
            break;
        case LV_EVENT_DELETE:
            // 对象删除时清理用户数据
            custom_data_t * data = lv_event_get_user_data(e);
            if(data) {
                lv_free(data);
            }
            break;
        default:
            break;
    }
}
```

**b) 用户数据管理**

- 通过 `lv_obj_set_user_data(obj, data)` 绑定的用户数据，必须在对象删除事件 (`LV_EVENT_DELETE`) 或页面销毁时手动释放。
- 对于复杂数据结构，建议实现统一的 `data_cleanup` 函数。





#### 3. 团队协作与代码质量控制

**a) 代码审查清单**
在合并 `generated/` 目录的自动生成代码时，必须检查：

- 控件命名是否符合规范（小写+下划线）
- 是否有重复的样式定义
- 事件回调是否正确绑定到 `event_init.c`
- 新增页面是否在 `custom.c` 中注册

**b) 资源文件管理**

- **图片资源**：建立图片命名规范，如 `{页面}_{控件}_{状态}.png`（`home_btn_icon_pressed.png`）
